# Azure DevOps Artifacts Interview Questions

## Most Frequently Asked Interview Questions

### Question 1

**Question**

What are Artifacts in Azure DevOps?

**Answer**

Artifacts are the output files generated during the build process that are stored and later used for deployment or package sharing.

Examples include:

- ZIP files
- JAR files
- WAR files
- DLL files
- Docker images
- Static website files
- NuGet packages
- npm packages

**Explanation**

Artifacts represent the deployable output of a successful build. Instead of rebuilding the application for every environment, the same artifact is promoted through Development, QA, UAT, and Production.

**Used in Production**

Artifacts are used by Release Pipelines, deployment pipelines, and package management systems.

**Common Mistake**

Many candidates confuse source code with artifacts. Source code is the input to the build process, while artifacts are the output.

---

### Question 2

**Question**

What are Build Artifacts?

**Answer**

Build Artifacts are files generated by a successful Build Pipeline and published for later use.

Examples:

- Compiled binaries
- Application packages
- Configuration files
- Deployment packages
- Test reports

A typical Build Pipeline workflow is:

```text
Source Code
      │
      ▼
Build
      │
      ▼
Run Tests
      │
      ▼
Publish Build Artifacts
      │
      ▼
Release Pipeline
```

**Explanation**

Build Artifacts ensure that the exact same application package is deployed to every environment.

**Used in Production**

Release Pipelines download Build Artifacts instead of rebuilding the application.

**Common Mistake**

Rebuilding the application during deployment instead of reusing published artifacts.

---

### Question 3

**Question**

What is Azure Artifacts?

**Answer**

Azure Artifacts is the package management service in Azure DevOps that hosts and shares software packages.

It supports package types such as:

- NuGet
- npm
- Maven
- Python
- Universal Packages

**Explanation**

Unlike Build Artifacts, Azure Artifacts stores reusable packages that can be consumed by multiple applications and teams.

**Used in Production**

Organizations host internal libraries, SDKs, and reusable packages in Azure Artifacts.

**Common Mistake**

Many candidates think Azure Artifacts and Build Artifacts are the same. Build Artifacts are deployment outputs, whereas Azure Artifacts is a package repository.

---

### Question 4

**Question**

What is the difference between Build Artifacts and Azure Artifacts?

**Answer**

| Build Artifacts | Azure Artifacts |
|-----------------|-----------------|
| Generated during a build | Package repository service |
| Used for deployments | Used for package management |
| Consumed by Release Pipelines | Consumed by applications and build pipelines |
| Usually short-lived | Supports long-term package versioning |

**Explanation**

Build Artifacts are deployment packages, while Azure Artifacts is designed for storing and sharing reusable software packages.

**Used in Production**

A Build Pipeline may publish Build Artifacts for deployment and also publish reusable libraries to Azure Artifacts.

**Common Mistake**

Using Azure Artifacts as a replacement for deployment artifacts.

---

### Question 5

**Question**

Why should Build Artifacts be published?

**Answer**

Publishing Build Artifacts allows:

- Deployment without rebuilding
- Consistent deployments
- Artifact versioning
- Easier rollback
- Reuse across multiple environments

**Explanation**

Publishing artifacts ensures that the exact validated build is available for deployment.

**Used in Production**

Every enterprise CI/CD pipeline publishes artifacts after a successful build.

---

### Question 6

**Question**

How are Build Artifacts published in Azure Pipelines?

**Answer**

Artifacts are published using the **Publish Build Artifacts** task or the **PublishPipelineArtifact** task in YAML.

Example:

```yaml
- task: PublishBuildArtifacts@1
  inputs:
    PathtoPublish: '$(Build.ArtifactStagingDirectory)'
    ArtifactName: 'drop'
```

**Explanation**

Publishing stores the build output so it can be downloaded by deployment pipelines.

**Used in Production**

Most Build Pipelines publish artifacts immediately after successful testing.

**Common Mistake**

Publishing source code instead of compiled build output.

---

### Question 7

**Question**

How are Build Artifacts consumed?

**Answer**

Build Artifacts are downloaded during deployment using artifact download tasks.

Typical workflow:

```text
Build Pipeline
      │
      ▼
Publish Artifact
      │
      ▼
Release Pipeline
      │
      ▼
Download Artifact
      │
      ▼
Deploy Application
```

**Explanation**

Deployment pipelines retrieve the published artifact and deploy it without rebuilding.

**Used in Production**

Artifacts are downloaded automatically before deployment begins.

---

### Question 8

**Question**

Why is it recommended to build once and deploy many times?

**Answer**

Building once provides:

- Consistency
- Predictable deployments
- Reliable testing
- Easier troubleshooting
- Reduced deployment risk

**Explanation**

The same tested artifact should be promoted through every environment to ensure identical application behavior.

**Used in Production**

This is a standard CI/CD best practice adopted by enterprise DevOps teams.

**Common Mistake**

Generating different builds for different environments.

Azure Artifacts supports:

- npm

### Question 10

**Question**

Why is artifact versioning important?

**Answer**

Artifact versioning provides:

- Traceability
- Rollback capability
- Dependency management
- Reproducible deployments
- Easier troubleshooting

**Explanation**

Each artifact version represents a specific build, making it possible to identify exactly what was deployed.

**Used in Production**

Organizations deploy versioned artifacts such as:

- Application v2.4.1
- API v1.8.0
- Library v5.0.2

**Common Mistake**

Overwriting artifacts instead of publishing versioned builds.

## Scenario-Based Interview Questions

### Scenario 1

**Question**

The Build Pipeline completes successfully, but the Release Pipeline cannot find the artifact. What would you check?

**Answer**

Verify:

1. Artifact publishing task executed successfully.
2. Artifact name matches the Release Pipeline configuration.
3. Build completed successfully.
4. Artifact retention policy.
5. Pipeline permissions.
6. Release Pipeline artifact linkage.

**Explanation**

Most deployment failures occur because artifacts were not published correctly or the Release Pipeline references the wrong artifact.

---

### Scenario 2

**Question**

Your organization rebuilds the application separately for Development, QA, and Production. Would you recommend this approach?

**Answer**

No.

The recommended approach is:

1. Build once.
2. Publish the artifact.
3. Deploy the same artifact to every environment.

**Explanation**

Building once eliminates inconsistencies and ensures the exact tested application reaches Production.

---

### Scenario 3

**Question**

A deployment fails because the downloaded artifact is corrupted. How would you troubleshoot?

**Answer**

Check:

- Build logs.
- Artifact publishing logs.
- Artifact download logs.
- Artifact size.
- Storage integrity.
- Build agent issues.

If necessary, rebuild the application and republish the artifact.

**Explanation**

Corrupted artifacts can result from interrupted publishing, storage issues, or incomplete downloads.

---

### Scenario 4

**Question**

Your team accidentally deletes an artifact required for a Production rollback. How can this risk be reduced?

**Answer**

Implement:

- Artifact retention policies.
- Backup procedures.
- Versioned artifacts.
- Controlled deletion permissions.
- Long-term storage for production releases.

**Explanation**

Retaining production artifacts enables quick rollback during incidents.

---

### Scenario 5

**Question**

Multiple teams need to share a common .NET library. Would you use Build Artifacts or Azure Artifacts?

**Answer**

Use **Azure Artifacts**.

Publish the reusable library as a versioned NuGet package so multiple applications can consume it.

**Explanation**

Azure Artifacts is designed for package sharing, while Build Artifacts are intended for deployments.

---

### Scenario 6

**Question**

A Release Pipeline deploys an older version of the application even though a new build completed successfully. What would you investigate?

**Answer**

Check:

- Selected artifact version.
- Release trigger configuration.
- Release Pipeline settings.
- Artifact alias.
- Build completion status.
- Manual artifact selection.

**Explanation**

Deployments can use an older artifact if the latest build is not correctly selected or linked.

---

### Scenario 7

**Question**

A developer modifies the application directly on the Production server instead of deploying a new artifact. Is this recommended?

**Answer**

No.

Changes should always be made in source code, built through the Build Pipeline, published as artifacts, and deployed using the Release Pipeline.

**Explanation**

Manual production changes bypass version control, reduce traceability, and make rollback difficult.

---

### Scenario 8

**Question**

Your manager wants every deployment to be traceable to a specific Build Pipeline run. How would artifacts help?

**Answer**

Each published artifact is associated with:

- Build number.
- Commit ID.
- Build logs.
- Pipeline execution.
- Build timestamp.

This creates complete traceability between deployments and source code changes.

**Explanation**

Artifact metadata simplifies auditing, troubleshooting, and compliance reporting.

---

### Scenario 9

**Question**

Your organization maintains reusable npm packages for internal projects. How would you manage them?

**Answer**

Publish the packages to **Azure Artifacts** and configure Build Pipelines to restore dependencies from the internal package feed.

**Explanation**

Azure Artifacts provides secure, centralized package management with version control and access management.

---

### Scenario 10

**Question**

A deployment fails because the required artifact version has expired and been deleted. What changes would you recommend?

**Answer**

- Increase artifact retention periods.
- Preserve production release artifacts.
- Archive critical releases.
- Implement retention policies based on environment and business requirements.

**Explanation**

Proper retention policies ensure that deployment and rollback artifacts remain available when needed.

---

### Scenario 11

**Question**

During a production incident, your manager asks how to identify exactly which application version is running. How would you determine this?

**Answer**

Use the deployment records and artifact metadata to identify:

- Artifact version.
- Build number.
- Commit ID.
- Deployment history.
- Pipeline execution logs.

This information allows you to trace the deployed application back to the exact source code and Build Pipeline execution.

**Explanation**

Artifact versioning and deployment history provide end-to-end traceability, enabling faster troubleshooting, auditing, and rollback decisions during production incidents.
